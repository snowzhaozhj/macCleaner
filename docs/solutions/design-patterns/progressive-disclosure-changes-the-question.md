# 渐进披露应切换用户问题，而非增加同类细节

## Context

渐进披露常被误解为“默认少显示一点，展开后把同类字段全部摊开”。这种切法只按信息密度分层，没有按用户正在解决的问题分层：折叠和展开仍要求用户完成同一种认知任务，只是后者更累。

更可复用的切法是按“用户问题／决策阶段”切层。第一层是**决策面孔**，用最少的信息帮助用户判断“值不值得做”；第二层是**审查面孔**，回答“这个对象到底是什么、为什么能这样处理、出错后如何恢复、我还能去哪里核对”。两层面对同一批对象，但信息职责不同，而不是字段数量不同。

PR #44 已将这个模式落到当前树：折叠分类行用分类名、体积占比、数量和体积支持快速判断（`crates/gui/frontend/src/lib/StreamingList.svelte:114`）；展开后则逐项显示完整路径、复制入口、体积、完整证据和 Finder 核对入口（`crates/gui/frontend/src/lib/StreamingList.svelte:127`）。这次实现是模式的一个实例，不是模式本身；可迁移的是“按问题换面孔”的分层方法。

这个模式还有一个容易被忽略的边界：从 GUI 通往 CLI、Finder、日志或管理后台的出口，也属于信息架构的一部分，必须准确表达目标界面实际支持的范围。当前 CLI 的全局选项只有 dry-run、跳过确认、永久删除和 JSON 输出（`crates/cli/src/main.rs:9`），`Clean` 子命令本身没有规则筛选参数（`crates/cli/src/main.rs:26`）。因此，界面可以诚实展示真实存在的 `mc clean`，但不能为了让出口看起来更精确而虚构 `--only`。

## Guidance

先为每一层写一句它必须回答的问题，再选字段和操作。若两层的问题写出来仍然相同，例如都是“有哪些文件”，就还没有完成渐进披露设计，只是做了密度切换。

折叠层应服务高频、低成本的去留判断：保留对象类别、相对规模、数量、风险摘要或推荐动作中真正改变决定的少数信号。它不是详情的缩略版，也不应塞入只有审查者才会解释的路径、证据链或技术术语。当前实例用分类占总命中体积的比例计算占比条（`crates/gui/frontend/src/lib/StreamingList.svelte:47`），并把数量与格式化体积放在同一分类头中（`crates/gui/frontend/src/lib/StreamingList.svelte:120`）。

展开层应换成审查任务所需的结构，而非简单解除截断。通常包括：稳定身份、完整位置、影响、恢复方式、来源或证据、可复制值，以及跳到权威系统核对的入口。当前实例在审查态传入 `full`，将 impact/recovery 从默认合并截断切换为逐行完整呈现（`crates/gui/frontend/src/lib/StreamingList.svelte:146`、`crates/gui/frontend/src/lib/EvidenceCard.svelte:24`）。这正是“换面孔”：信息的组织方式随任务改变。

两层之间应保持业务语义不变。展开可以增加核对、复制、定位和选择能力，但不应暗中改变默认选择、风险分级、确认门槛或最终动作。若审查信息会实质改变安全判断，它就不能只藏在展开层；不可逆动作所需的关键风险必须在行动前显式出现。

跨界面出口遵循三条诚实原则：

1. **能力真实。** 只展示目标界面当前确实接受的命令、参数或深链；目标能力不存在时，宁可不展示。
2. **范围真实。** 出口无法精确映射当前项时，必须标明它作用于更宽范围，不能把“整体操作”包装成“当前项操作”。当前实现只呈现一次 `mc clean`，并明确标注“清理全部可安全释放项”（`crates/gui/frontend/src/lib/StreamingList.svelte:80`）。
3. **失败真实。** 外部目标可能已消失或不可达，失败必须可见且不改变原操作状态。当前 Finder 出口通过 IPC 传递原始路径（`crates/gui/frontend/src/lib/ipc.ts:177`），后端在路径不存在或系统命令失败时返回错误（`crates/gui/src/commands/reveal.rs:15`），前端再将错误显示为提示而非静默吞掉（`crates/gui/frontend/src/lib/StreamingList.svelte:68`）。

复制和跳转最好抽成很小的呈现原语，但不要让原语替业务语义做决定。当前复制按钮只负责写入传入文本并提供成功／失败的短暂反馈（`crates/gui/frontend/src/lib/CopyButton.svelte:18`）；命令是否真实、作用范围如何描述，仍由使用它的界面负责。

验证时同时覆盖“看起来正确”和“语义确实正确”：

- 折叠态验证无需展开即可看见决定所需信号，并检查视觉元素预算没有膨胀。
- 展开态验证完整身份、完整证据、核对入口和复制入口；不能只断言元素存在，还应断言内容未被错误替换或缩窄。
- 跨界面出口以目标系统的解析器或 API 契约为权威，增加对命令名、参数和作用范围文案的断言；不存在的参数应有负向守卫。
- 外部跳转验证调用参数与当前项一致，并覆盖目标不存在、权限失败等降级路径。当前 E2E 已验证页面展示 `mc clean`、展开后出现完整路径，以及 Finder 调用携带该路径（`crates/gui/frontend/e2e/clean.spec.ts:164`）。
- 安全不变量单独保留回归测试。当前 Clean E2E 使用未选中的 Risky fixture，验证手动选择并输入确认口令后才提交（`crates/gui/frontend/e2e/clean.spec.ts:77`）；默认预选语义需要由独立契约测试保护，不能从该用例反推。

## Why This Matters

按问题分层会降低两类认知成本。普通用户不会被实现细节迫使成为审查者，开发者也不会因为默认界面简化而失去核对能力。展开动作因此有清晰收益：它不是“显示更多”，而是进入另一种工作模式。

这种分层也能阻止界面在迭代中退化成 badge、字段和按钮的堆积。每个新增元素都必须回答一个可检验的问题：它帮助第一层做决定，还是帮助第二层做审查？两者都不是，就不应占据这两层。

诚实出口维护的是用户对整个产品的信任。一个看似便利但目标系统不支持的参数，会把界面的承诺变成运行时失败；一个真实但范围被误标的命令，则可能执行比用户理解的更宽的动作。相比之下，明确展示较宽但真实的出口，或暂时不提供精确出口，都是更安全的产品行为。

不过，渐进披露不是隐藏复杂性的通行证。若第一层缺少做出安全决定的必要信息，或第二层只是把相同列表变长，这个模式就失败了。正确的标准不是默认界面有多简洁，而是每一层能否独立完成它承诺的问题。

## When to Apply

- 同一对象同时服务快速决策者和深度审查者；或同一用户需要先扫视候选集，再审查少数对象。
- 默认流程频繁而详情读取低频，详情又具有明显的身份、证据、后果或恢复结构。
- GUI 需要把高级用户送往 CLI、Finder、日志平台等权威工具继续工作。
- 目标系统只能处理整体而当前界面展示单项时，只有在范围被清楚标明后才提供整体出口。

以下情况不适用：关键风险信息必须在动作前始终可见；展开后的内容无法形成独立审查任务；对象太少、信息太短，分层反而增加点击成本；或目标系统没有稳定、可验证的出口。此时应直接展示必要信息、重新组织主界面，或省略跨界面操作。

## Examples

**正确的两层。** 折叠层显示“系统缓存、占总量约六成、2 项、8 MiB”，回答“这一类值不值得清”；展开后显示每一项的完整路径、影响、恢复方式、复制和 Finder 定位，回答“我要删的到底是什么”。当前代码分别在分类头中构造占比与元数据（`crates/gui/frontend/src/lib/StreamingList.svelte:114`），在展开审查行中构造逐项身份与证据（`crates/gui/frontend/src/lib/StreamingList.svelte:127`）。

**错误的密度切层。** 折叠层显示前三条路径，展开层显示全部路径；或折叠时截断 impact/recovery，展开时只解除截断，却没有完整身份、核对入口和恢复语义。两层都在回答“有哪些条目”，只是信息量不同，应重新定义第二层问题。

**诚实的 CLI 出口。** 当 CLI 只支持整体 `Clean` 时，界面展示 `mc clean` 并说明它清理全部可安全释放项；当前命令入口和范围标签就是这样配对的（`crates/gui/frontend/src/lib/StreamingList.svelte:82`）。不要展示 `mc clean --only system-cache`，因为当前 `Clean` 变体不接受这样的字段（`crates/cli/src/main.rs:27`）。将来只有在 CLI 解析器、稳定标识和 GUI→CLI 映射都存在后，才能把精确命令标成等价出口。

**诚实的外部核对。** “在 Finder 中显示”应把当前项完整路径原样交给后端（`crates/gui/frontend/src/lib/StreamingList.svelte:148`），后端只执行定位并返回成败（`crates/gui/src/commands/reveal.rs:20`）。路径已经消失时，界面报告失败，不伪装成已打开，也不顺带改变选择或删除状态。

**验证模板。** 为一个新场景至少写五类断言：折叠层能完成决定；展开层换成审查结构；复制内容与可见内容一致；跨界面调用的命令和参数真实；不可逆动作的既有安全门槛不变。再加一条负向断言，确保未支持的精确命令、深链或筛选参数不会被渲染。这样测试保护的是产品语义，而不仅是 DOM 形状。

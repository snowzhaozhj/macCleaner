# TUI 交互体验全面修复

## 概述

四个功能（Clean、Uninstall、Analyze、Purge）存在阻断级 bug 和交互不统一问题。分两阶段修复：第一阶段修阻断级问题确保基本可用，第二阶段统一交互模型提升一致性。

## 第一阶段：阻断级修复

### P5: Uninstall 应用扫描冻结 UI

- `AppResolver::list_apps()` 在主线程同步执行，TUI 完全冻结
- 其他三个功能都有后台线程 + 扫描进度反馈
- **目标：** 改为后台线程 + 显示 Scanning 状态和 spinner

### P2/P12: 清理过程无法取消

- Cleaning 状态不处理任何按键事件
- 用户只能 Ctrl+C 杀掉整个程序
- **目标：** Esc 取消当前清理（已清理的不回滚），显示部分完成结果

### P7: Analyze 删除标记失效

- `d` 键标记路径到 `marked_for_delete` 集合
- 但没有任何流程触发实际删除
- 退出时标记被静默丢弃
- **目标：** 要么加一个确认删除流程（Enter/Delete 键触发），要么去掉 `d` 键功能避免误导

### P10: Sorting 阶段取消丢失所有数据

- 排序是一个短暂过渡状态（通常 < 1 秒）
- 但按 q/Esc 会丢弃整个扫描树
- **目标：** Sorting 期间不响应 q/Esc，或只显示不可交互的 spinner

## 第二阶段：交互统一

### P11: 统一返回/退出键位

当前混乱状态：
- Menu: q → 退出程序
- Results: q → 返回菜单
- Analyzing: q → 返回菜单，Esc → 返回上级/中止
- Done: q → 退出程序，Enter → 返回菜单

**目标：定义统一规则**
- `q` / `Esc` 始终返回上一层（Results → Menu, Analyzing 子目录 → 父目录）
- 在 Menu 层 `q` 退出程序
- 取消当前操作统一用 `Esc`

### P9: Analyze 中 Esc 语义过载

- 在子目录中 Esc = 返回上一级（合理）
- 在根目录中 Esc = 中止扫描并 finalize（不合理，用户预期是返回菜单）
- **目标：** 根目录 Esc 弹出确认"停止扫描并返回菜单？"

### P1: 扫描中不能提前操作

- 用户在 Scanning 状态可以浏览和选择，但不能按 Enter 确认
- **目标：** 允许扫描中按 Enter 确认已选项（扫描继续在后台跑或中止）

### P8: Analyze 直播阶段排序

- 当前按 DFS 遍历顺序显示，用户看到的是乱序
- **目标：** 实时阶段也按已知大小排序（或至少提供排序切换键）

### P3/P4: 扫描信息整理

- 进度信息有重复（已扫描数 vs 已发现数）
- 列表标题硬编码未区分功能
- **目标：** 精简进度显示，标题反映当前功能名

## 非需求

- 不改变 TUI 的整体布局框架（上中下三栏）
- 不新增功能——只修复和统一现有功能
- 不做 GUI

## 成功标准

- 四个功能都不会冻结 UI
- 所有长时间操作都可取消
- 返回/退出操作在四个功能中行为一致
- Analyze 的标记删除功能要么完整可用要么不存在
